# ⚠️ 上线前必须执行的安全修复

## 🔒 当前安全问题

**当前状态**: ❌ **数据库允许任何人写入数据（不安全）**

这意味着：
- 任何人都可以访问 `/admin` 页面
- 即使没有登录，也可以修改数据库
- 恶意用户可能删除或篡改数据

## ✅ 解决方案

### 步骤 1: 执行 RLS 策略更新 SQL

**必须执行以下 SQL 来修复安全问题**：

1. **登录 Supabase Dashboard**
   - 访问：https://app.supabase.com
   - 选择你的项目

2. **打开 SQL Editor**
   - 左侧菜单 → `SQL Editor`
   - 点击 `New query`

3. **复制并执行以下 SQL**：

```sql
-- ============================================
-- 更新 RLS 策略：要求认证才能写入
-- ============================================

-- 1. Speakers 表
DROP POLICY IF EXISTS "Allow public write access" ON speakers;
CREATE POLICY "Allow authenticated write access" ON speakers
  FOR ALL USING (auth.role() = 'authenticated');

-- 2. Sponsors 表
DROP POLICY IF EXISTS "Allow public write access" ON sponsors;
CREATE POLICY "Allow authenticated write access" ON sponsors
  FOR ALL USING (auth.role() = 'authenticated');

-- 3. Agenda Items 表
DROP POLICY IF EXISTS "Allow public write access" ON agenda_items;
CREATE POLICY "Allow authenticated write access" ON agenda_items
  FOR ALL USING (auth.role() = 'authenticated');

-- 4. FAQ Items 表
DROP POLICY IF EXISTS "Allow public write access" ON faq_items;
CREATE POLICY "Allow authenticated write access" ON faq_items
  FOR ALL USING (auth.role() = 'authenticated');

-- 5. Universities 表（如果存在）
DROP POLICY IF EXISTS "Allow public write access" ON universities;
CREATE POLICY "Allow authenticated write access" ON universities
  FOR ALL USING (auth.role() = 'authenticated');

-- 6. Contact Links 表（如果存在）
DROP POLICY IF EXISTS "Allow public write access" ON contact_links;
CREATE POLICY "Allow authenticated write access" ON contact_links
  FOR ALL USING (auth.role() = 'authenticated');

-- 7. Hero Sections 表（如果存在）
DROP POLICY IF EXISTS "Allow public write access" ON hero_sections;
CREATE POLICY "Allow authenticated write access" ON hero_sections
  FOR ALL USING (auth.role() = 'authenticated');

-- 8. Posters 表（如果存在）
DROP POLICY IF EXISTS "Allow public write access" ON posters;
CREATE POLICY "Allow authenticated write access" ON posters
  FOR ALL USING (auth.role() = 'authenticated');
```

4. **点击 `Run` 执行 SQL**

5. **确认执行成功**
   - 应该看到 "Success. No rows returned" 或类似消息
   - 如果有错误，检查错误信息并修复

### 步骤 2: 验证修复

**测试方法**:

1. **未登录状态测试**:
   - 打开浏览器隐私模式（或退出登录）
   - 访问 `https://你的域名.com/admin`
   - 尝试添加一个 Speaker
   - **应该失败**（显示错误或要求登录）

2. **登录状态测试**:
   - 使用 Supabase Auth 账号登录
   - 尝试添加一个 Speaker
   - **应该成功**

### 步骤 3: 创建管理员账号（如果还没有）

**必须至少创建一个 Supabase Auth 管理员账号**：

1. **在 Supabase Dashboard**
   - `Authentication` → `Users`
   - 点击 `Add user` → `Create new user`
   - 填写：
     ```
     Email: 你的邮箱
     Password: 强密码（至少 12 位）
     Auto Confirm User: ✅ 勾选
     ```
   - 点击 `Create user`

2. **测试登录**
   - 访问 `/admin`
   - 使用邮箱和密码登录
   - 确认可以正常使用

## ⚠️ 重要提醒

### 执行 SQL 后的影响

**✅ 好处**:
- 只有认证用户才能修改数据
- 提高安全性
- 可以追踪操作者

**⚠️ 注意事项**:
- 未登录用户无法修改数据（这是正常的）
- 必须使用 Supabase Auth 登录才能使用 Admin 页面
- 如果使用简单密码验证，需要先登录 Supabase Auth

### 如果执行后出现问题

**问题**: 登录后无法保存数据
**解决**: 
1. 检查用户是否已认证（Supabase Dashboard → Authentication → Users）
2. 检查 RLS 策略是否正确执行
3. 检查浏览器控制台错误信息

**问题**: 执行 SQL 时报错
**解决**:
- 如果表不存在，跳过该表的 SQL
- 如果策略不存在，`DROP POLICY` 会失败，可以忽略
- 检查错误信息，根据提示修复
